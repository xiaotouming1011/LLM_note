```python

最终交付物
硬性约束
隐含的要求
什么样才算完成
最容易犯错误的地方

要求：
尽量简短
不啊偷偷做实验假设
不确定的地方请明确标出来
写入markdown文档:任务大纲.md
```

2.给出方案

```python
现在不要写代码。
对于刚才已经确认的问题定义，给我3条可行方案
每条方案都讲清楚：大致怎么做，主要优点，主要风险，实现复杂度，可测试性
最后请明确推荐一条方案，并说明为什么不选择另外几条。要求优先考虑稳定交付，而不是最炫。沉淀到markdown文档里：方案.md
```

3.先定骨架（模块，接口，数据）再开始写

```python
对于已经选定的方案,现在先不要给展开实验细节。
先给出一份足够支付编码开始的系统骨架，至少包括：
模块拆分
目录结构
各模块职责
核心借口或API
关键数据结构
主链路时序
权限，隔离和边界

要求：
目标“足够支撑编码”
不要写成长篇设计文档
如果边界还不稳，请明确标出来
```

```python
现在开始实现，但只做最小可运行闭环

  应该按照什么顺序实现
  那些东西现在必须做
  哪些东西现在明确先不做

后面只围绕这个最小闭环开始写代码
要求：
	不要提前铺满所有功能
  每完成一部分，都给出最短验证方式
  如果发现规格有冲突，先指出，不要默默改掉
```

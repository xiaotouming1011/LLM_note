# 六大核心中间件 / 数据库 

按**日常开发权重 + 理解难度**排序，每一个讲清：**核心作用、解决什么问题、适用场景、本质区别**，看完彻底不混淆。

## 1. Redis

**定位：高性能内存缓存 + 轻量中间件**

- 本质：数据存在**内存**里，读写速度极快（毫秒级）

- 核心干什么：

  1. 缓存热点数据（商品详情、首页数据、用户登录信息），减轻数据库压力
  2. 做限流、分布式锁、计数器、排行榜、秒杀库存
  3. 临时存短生命周期数据（验证码、token、临时会话）

  

- 特点：快、支持多种数据结构、可持久化；**不适合存海量永久数据**

- 一句话：**一切要快、要高频读写、临时存储的场景，全用 Redis**

## 2. MySQL

**定位：关系型数据库，企业核心业务存储**

- 本质：结构化表格存储，有行 / 列、主键、外键、事务

- 核心干什么：

  1. 存**核心永久业务数据**（用户账号、订单、支付、商品、账单）
  2. 需要严格数据一致性、事务（下单扣库存 + 扣款要么都成、要么都失败）
  3. 复杂联表查询、统计、权限管控

  

- 特点：数据落地硬盘、稳定、强一致性、不能扛超高并发读写

- 一句话：**公司最重要、不能丢的核心数据，全部存在 MySQL**

## 3. MongoDB

**定位：非关系型文档数据库（NoSQL）**

- 本质：存**JSON 格式文档**，没有固定表结构，灵活自由

- 核心干什么：

  1. 结构经常变的业务（用户画像、动态表单、小程序自定义数据）
  2. 海量弱结构化数据、日志、内容草稿、评论信息流
  3. 不需要复杂联表、追求写入灵活的场景

  

- 特点： schema 自由、写入快、扩容简单；**不适合强事务、复杂联表**

- 一句话：**数据结构乱七八糟、经常改字段，就用 MongoDB**

## 4. RabbitMQ

**定位：轻量业务级消息队列**

- 本质：消息中转站，异步削峰、解耦

- 核心干什么：

  1. 系统解耦：下单后不用同步发短信 / 物流，扔给队列，后台慢慢消费
  2. 流量削峰：秒杀、活动大流量，排队慢慢处理，不压垮数据库
  3. 可靠投递：保证消息不丢、业务重试、死信队列

  

- 特点：侧重**业务可靠性、稳定性、易用**，吞吐量中等

- 一句话：**普通业务异步、发短信、通知、工单，用 RabbitMQ**

## 5. Kafka

**定位：大数据级高吞吐消息队列**

- 本质：分布式日志 / 消息流引擎，极致高吞吐

- 核心干什么：

  1. 海量日志采集（用户行为日志、操作日志、埋点数据）
  2. 大数据实时计算、流式数据传输、亿级消息积压
  3. 高并发削峰、跨机房数据同步

  

- 特点：**吞吐爆炸高、适合海量数据流**，业务可靠性弱于 RabbitMQ

- 一句话：**海量日志、大数据、超高流量数据流，必用 Kafka**

## 6. Elasticsearch（ES）

**定位：全文搜索引擎 + 海量日志检索分析**

- 本质：倒排索引，专门做**模糊搜索、全文检索**

- 核心干什么：

  1. 全站搜索（淘宝商品搜索、知乎文章搜索、文档检索）
  2. 海量日志统一收集、检索、告警（运维日志、系统报错日志）
  3. 大数据模糊查询、分词搜索、地理位置检索

  

- 特点：模糊查询吊打数据库，不适合存核心业务数据

- 一句话：**只要需要模糊搜索、全文检索、日志排查，就用 ES**

------

# 极简场景对照（面试 / 工作直接背）

1. 核心业务存数据 → **MySQL**
2. 要快、缓存、秒杀、分布式锁 → **Redis**
3. 数据结构灵活多变 → **MongoDB**
4. 业务异步解耦、短信 / 通知 → **RabbitMQ**
5. 海量日志、埋点、大数据流 → **Kafka**
6. 全文搜索、日志检索 → **Elasticsearch**

------

# 补充：Python 操控学习优先级（最合理）

Redis → MySQL → MongoDB → RabbitMQ → Kafka → ES

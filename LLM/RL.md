#### 强化学习



从 `策略梯度` 到 `TRPO`、`PPO`、`DPO`、`GRPO`，本质上是在不断解决这几个痛点：

- 训练不稳定
- 更新太猛，模型容易崩
- 需要复杂的价值函数/奖励模型
- 样本效率低
- 对齐人类偏好太麻烦
- 训练成本太高

---

## 1. 策略梯度：最原始的想法
先把它想成：

**模型每次做一个动作，环境给一个奖励；做得好就强化，做得差就削弱。**

在大模型里，动作就是：
- 生成下一个 token
- 生成一整段回答

策略梯度的核心是：
- 直接优化“策略”，也就是模型输出分布
- 让高奖励的输出概率变大，低奖励的输出概率变小

### 优点
- 思路直接
- 能处理连续动作或大动作空间
- 很适合“生成式模型”

### 问题
- **方差大**：训练信号很抖
- **更新不稳**：一次改太多，模型容易跑偏
- **样本效率差**：要很多试错
- **需要 baseline / critic** 来降方差，不然太难训

### 形象比喻
像一个新手司机，方向盘一把打太猛：
- 可能拐对了
- 也可能直接冲出路

所以大家开始想：**能不能更新得别那么猛？**

---

## 2. TRPO：先保证“别乱改”
TRPO 的核心想法是：

**可以更新策略，但每次只能小步走，不能把老策略一下子改崩。**

它引入了一个很重要的约束：
- 新策略不能离旧策略太远
- 用 KL 散度限制更新幅度

### 为什么需要 TRPO
因为策略梯度虽然能学，但太容易“学过头”：
- 今天这个回答高奖励，概率拉太高
- 结果其它能力一起被破坏
- 训练很不稳定

TRPO 解决的是：
- **稳定性**
- **单次更新别太激进**

### 但 TRPO 的问题
TRPO 虽然稳，但它很麻烦：
- 优化过程复杂
- 计算开销大
- 实现不方便
- 工程上不够友好

### 形象比喻
策略梯度是“猛打方向盘”  
TRPO 是“给方向盘加限位器”  
这样不会一下子拐飞，但开起来太复杂、太重。

所以大家继续想：**能不能保留“别乱改”，但让它更好实现？**

---

## 3. PPO：TRPO 的工程简化版
PPO 的价值很大，因为它基本把强化学习训练真正拉进了工程可用阶段。

它的核心想法是：

**不要真的做很复杂的约束优化，而是用一个“夹子”把策略更新限制住。**

PPO 常说的 `clipping` 就是这个意思：
- 如果新策略变化太大，就把它“截住”
- 避免一步走太远

### 为什么 PPO 比 TRPO 更重要
因为它：
- 更简单
- 更容易实现
- 更稳定
- 效果通常也很好

### PPO 解决了什么
- 保留了“别更新太猛”的思想
- 去掉了 TRPO 的复杂约束求解
- 成为 RLHF 的主力工具

### PPO 仍然有的问题
它还是比较“重”：
- 通常要训练 reward model
- 常常还要训练 value model / critic
- 训练链条长
- 工程复杂
- 成本高
- 调参麻烦

### 形象比喻
TRPO 像“高级汽车防侧翻系统”  
PPO 像“普通车上更实用的电子稳定系统”  
还是有约束，但更轻便、更能落地。

---

## 4. DPO：绕开 PPO 的复杂链路
DPO 的出现，核心是想问一句：

**既然我们最终想要的是“更符合人类偏好”，为什么一定要先训练一个奖励模型，再用 PPO 去优化？**

DPO 的想法是：
- 不走“奖励模型 + 强化学习”那条重链路
- 直接用偏好数据优化模型

也就是：
- 人类给你两个回答：A 比 B 更好
- 模型直接学：以后更偏向 A，不偏向 B

### 为什么需要 DPO
因为 PPO 那条路太重：
- 奖励模型要训练
- PPO 还要训练
- 中间环节多
- 不稳定
- 代价高

DPO 的价值是：
- **更简单**
- **更稳定**
- **更容易训**
- **更像监督学习**
- 对齐人类偏好更直接

### 形象比喻
PPO 像：
- 先请一个裁判打分
- 再让选手根据裁判分数反复调整

DPO 像：
- 直接告诉选手：这个答案比那个答案好
- 你直接学偏好就行

### DPO 的代价
它简化了训练，但也有边界：
- 很依赖高质量偏好数据
- 它不是完全替代所有 RL 方法
- 对某些需要复杂探索的任务，未必最强

---

## 5. GRPO：继续降低训练负担
GRPO 的目标可以理解成：

**我还想保留“用偏好/奖励来优化”的效果，但能不能把训练再简化一点，尤其不要太依赖 value model？**

它通常被理解为一种更轻量的 group-based 相对优化思路，核心是：
- 同一个 prompt 生成多个候选
- 在组内比较谁更好
- 用相对优势来更新
- 尽量减少对单独 value model 的依赖

### 为什么会有 GRPO
因为即便 DPO 简化了流程，很多场景仍然有需求：
- 想要强化学习式优化
- 但不想再背那么重的 critic/value 负担
- 希望训练更稳、更省

### GRPO 的思想
不是单独看一个回答绝对好不好，而是：
- 在一组回答里比较
- 谁相对更好就强化谁
- 用 group 相对优势来更新

### 形象比喻
PPO 是“给每个回答单独打分，再更新”  
GRPO 是“把一组回答放一起比，比出来谁更强，再更新”

这样做的好处是：
- 训练结构更轻
- 常常更省资源
- 对一些大模型对齐任务更实用

---

# 一条线看懂
你可以把整个演化看成：

### 策略梯度
“直接学，但容易乱”

### TRPO
“别乱学，更新步子受限，但太复杂”

### PPO
“还是别乱学，但用更简单的方式限制步子，工程上好用”

### DPO
“既然目标是人类偏好，干脆直接学偏好，不绕奖励模型 + 强化学习那一大圈”

### GRPO
“还能不能更轻一点？别太依赖 value/critic，用组内相对比较来优化”

---

# 为什么不断换策略
本质就是在不断换一个更好的折中：

- **稳定性**
- **训练复杂度**
- **算力成本**
- **样本效率**
- **对齐效果**
- **工程可落地性**

每一代方法都在回答同一个问题：

**怎么在不把模型训崩的前提下，让它更像人想要的样子？**

---

# 如果用一句话概括
- `策略梯度`：能学，但太抖
- `TRPO`：稳，但太重
- `PPO`：稳且好用，成为主流
- `DPO`：直接学偏好，省掉很多复杂环节
- `GRPO`：进一步轻量化相对优化，减少训练负担

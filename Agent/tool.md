## **工具调用的三大失败模式**

### **失败模式一：参数提取错误**

用户说"帮我查后天的天气"，LLM 需要把"后天"转换成具体日期。如果今天是 4 月 28 日，后天就是 4 月 30 日。但 LLM 可能算错，传了 5 月 1 日，导致查询结果不对。

更常见的是参数缺失。用户说"帮我查天气"，没说城市，LLM 可能瞎猜一个城市（比如默认北京），也可能返回错误"缺少必需参数 city"。

解决思路：在工具定义中明确参数的格式和约束，比如 date 参数必须是 YYYY-MM-DD 格式。对于缺失的参数，让 LLM 主动追问用户，而不是瞎猜。



### **失败模式二：工具选择错误**

用户说"帮我订一张明天去上海的机票"，系统里有两个工具：search_flights（搜索航班）和 book_flight（预订航班）。LLM 可能直接调用 book_flight，但这个工具需要航班号参数，用户没提供，调用失败。

正确的流程应该是：先调用 search_flights 搜索可用航班，展示给用户选择，用户选定后再调用 book_flight 预订。但 LLM 可能跳过第一步，直接尝试预订。

解决思路：在工具描述中明确工具的前置条件和使用场景。比如 book_flight 的描述里写"需要先调用 search_flights 获取航班号"。也可以用 ReAct 框架，让 LLM 在每次工具调用前先输出推理过程（Thought），提高决策质量。



### **失败模式三：工具执行超时或返回异常**

工具调用了外部 API，API 可能超时、返回 500 错误、返回空结果。普通程序遇到这种情况会抛异常，但 Agent 需要把错误信息返回给 LLM，让 LLM 决定怎么办。

比如调用 get_weather 返回"API 超时"，LLM 可能会：

- 重试一次
- 换个天气 API
- 告诉用户"抱歉，天气服务暂时不可用，请稍后再试"

但如果错误信息不够清晰，LLM 可能做出错误决策。比如 API 返回"500 Internal Server Error"，LLM 可能不知道这是临时故障还是永久故障，盲目重试多次浪费时间。

解决思路：对工具执行结果做标准化处理，把各种错误码转换成 LLM 能理解的自然语言描述。比如"API 超时"转换成"天气服务响应超时，建议重试或使用备用服务"。



# 你的multi-agent系统有没有遇到过json生成错误，生成不完整的问题？

**遇到过，而且这是 Multi-Agent 系统里非常典型、必须系统性解决的问题。**

在我们的多智能体系统中，Agent 之间是通过结构化 JSON 传递意图、工具调用参数、以及下一跳路由信息的。

在实际运行中，确实频繁出现两类问题：

1. **JSON 语法错误**：引号不匹配、单引号、尾逗号、中文标点、被 markdown 包裹等。
2. **JSON 生成不完整**：模型输出被截断、思考过程太长导致后半段被切掉、字段缺失。

这些问题会直接导致 Agent 之间通信中断、工具调用失败、整个任务链路卡住。

所以我们从**工程层面做了一整套容错机制**，主要分为四层：

------

# 1. 输出约束层：从源头减少错误

在 Prompt 里强制模型：

- 只输出 JSON，不允许任何自然语言描述
- 严格遵循固定 Schema
- 明确禁止使用单引号、中文标点、尾逗号
- 必要时使用少量 Few-shot 示例

这一步能显著降低错误率。

------

# 2. 清洗解析层：修复常见错误

模型输出后，不会直接 `json.loads`，而是先做**结构化清洗**：

- 去除 ````json` 包裹
- 提取最外层 `{ ... }`
- 单引号转双引号
- 去除尾逗号
- 修复中文标点
- 尝试容错解析

大部分不规范 JSON 都能在这里被修复。

------

# 3. Schema 校验层：保证结构合法

修复后使用 **Pydantic** 做严格校验：

- 字段是否存在
- 类型是否正确
- 必填字段不能为空
- 枚举值是否合法（比如 next_agent、action 类型）
- 参数格式是否符合工具要求

只要校验不通过，就不会进入执行流程。

------

# 4. 重试与兜底层：保证系统高可用

如果解析或校验失败，系统不会直接崩溃，而是：

- **自动重试 2～3 次**，告诉模型 “格式错误，请重新严格输出 JSON”

- 仍然失败则进入**降级策略**：

  - 返回默认空动作
  - 交给调度 Agent 重新规划
  - 或记录错误后跳过当前步骤，保证主流程不中断

  

同时我们会把所有 JSON 失败日志埋点，用于后续优化 Prompt 和模型选择。

------

# 总结句（面试官最爱）

**简单说，我们没有依赖模型本身保证格式稳定，而是通过 “强提示 + 清洗修复 + Schema 校验 + 重试兜底” 四层机制，把 JSON 错误对系统的影响降到了最低，保证了 Multi-Agent 系统在生产环境的稳定性。**

遇到过，主要是 JSON 语法错误和截断不完整。

我们的处理方式是：

1）提示词强制输出规范 JSON；

2）做统一的清洗与容错解析；

3）用 Pydantic 做 Schema 校验；

4）失败自动重试，最终兜底保证流程不中断。

通过这套工程化机制，基本解决了 JSON 不稳定的问题。

#### 1.Agent 的记忆分哪三种类型？各自用什么存储方式？

语义记忆：通用知识（产品条款/政策/FAQ），对所有用户共享，存共享向量数据库，按内容语义检索

情节记忆：用户专属历史信息（偏好/资质/历史决策），按 user_id 隔离存 Milvus，检索时先过滤 user_id 再语义检索

程序性记忆：操作流程和行为规则（'涉及高净值客户优先推荐私行产品'），直接固化为 System Prompt，不需要检索

短期记忆：对话上下文窗口，保留N轮对话

```python
class ShortTermMemory:
     def __init__(self, window_size: int = 10):
         self.window_size = window_size
         self.messages = []

     def add_message(self, role: str, content: str):
         self.messages.append({"role": role, "content": content})
         # 超出窗口大小则移除最早的消息
         if len(self.messages) > self.window_size * 2:  # 每轮包含user+assistant两条
             self.messages = self.messages[-self.window_size * 2:]

     def get_context(self) -> list:
         return self.messages
```

### Generative Memory论文：重要性评分，反思机制

```python
# 基于Generative Agents论文的检索公式
 def compute_retrieval_score(
     memory: dict,
     query_embedding: list,
     current_time: float,
     decay_rate: float = 0.995
 ) -> float:
     """
     综合评分 = α*语义相关性 + β*时间新鲜度 + γ*重要性
     """
     # 语义相关性：query与memory的embedding余弦相似度
     semantic_score = cosine_similarity(query_embedding, memory["embedding"])

     # 时间衰减：记忆越久远，分数越低
     hours_passed = (current_time - memory["created_at"]) / 3600
     recency_score = decay_rate ** hours_passed

     # 重要性：LLM给这条记忆打的重要性分数（1-10，归一化到0-1）
     importance_score = memory["importance_score"] / 10.0

     # 三项加权求和（权重可调）
     alpha, beta, gamma = 0.4, 0.3, 0.3
     return alpha * semantic_score + beta * recency_score + gamma * importance_score
```

**反思机制（Reflection）：** 当记忆库里积累的重要性分数总和超过某个阈值（比如 150 分），系统触发一次反思——让 LLM 对已有记忆进行更高层次的抽象和总结，生成"高阶记忆"写入记忆库。

## **MemGPT：虚拟内存管理的工程启示**

同年 UC Berkeley 发表的 MemGPT 论文，核心思想是把操作系统的虚拟内存管理概念引入 LLM：主存（上下文窗口）容量有限，需要换入换出机制来处理"无限"的外部存储。

它的关键创新是**自管理机制**：LLM 通过函数调用自主决定何时读写记忆，而不是由外部系统被动触发。这意味着 Agent 本身有能力感知"我的上下文快满了，需要把某些信息存到长期记忆里"，而不是到了窗口上限才被系统强制截断。

这个思想对我们的工程设计有一个直接影响：我们给 Agent 提供了两个工具函数——`store_memory(content)` 和 `recall_memory(query)`，让 LLM 在生成回答的过程中自主决定何时调用它们，而不是在对话结束后统一批量处理。

## **记忆冲突怎么处理**

**方案一：语义去重判断**

用 LLM 比较新旧信息的语义相似度，相似度超过阈值则判断为同一实体的更新

```python
# 记忆冲突检查逻辑（异步版）
用于判断新记忆是 ADD 新增，还是 UPDATE 更新


async def check_memory_conflict(
    new_memory: str,
    existing_memories: list,
    similarity_threshold: float = 0.85
) -> dict:
    """
    检查新记忆是否与现有记忆冲突
    返回：{"action": "ADD/UPDATE", "conflict_memory_id": str or None}
    """
    # 无历史记忆 → 直接新增
    if not existing_memories:
        return {"action": "ADD", "conflict_memory_id": None}

    # 生成新记忆的向量
    new_embedding = await embedder.aembed_query(new_memory)

    # 遍历所有现有记忆，计算语义相似度
    for mem in existing_memories:
        similarity = cosine_similarity(new_embedding, mem["embedding"])

        # 相似度超过阈值 → 判定为同一条记忆
        if similarity > similarity_threshold:
            # 调用 LLM 二次确认：是否属于内容更新
            is_update = await llm_confirm_update(new_memory, mem["content"])
            if is_update:
                return {"action": "UPDATE", "conflict_memory_id": mem["memory_id"]}

    # 无冲突/无相似 → 新增
    return {"action": "ADD", "conflict_memory_id": None}
```

**方案二：记忆过期与 TTL 管理**

"用户本月预算是 5 万"这类信息，30 天后就过期了，继续保留反而会干扰后续回答。TTL 机制让系统自动完成清理，不需要人工干预。

```python

     content: str,
     user_id: str,

 ):

         ttl_timestamp = int(time.time()) + ttl_days * 86400

     memory_record = {
         "memory_id": str(uuid.uuid4()),
         "user_id": user_id,
         "content": content,
         "created_at": int(time.time()),
         "ttl": ttl_timestamp,
         "is_deleted": False
     }
     milvus_client.insert("agent_memory", memory_record)

 # 示例：本月预算信息，有效期30天

 async def cleanup_expired_memories():

     expired = milvus_client.query(
         collection_name="agent_memory",
         filter=f"ttl > 0 && ttl < {current_time} && is_deleted == false"
```

## **进阶：混合检索公式——不只是语义相似度**

从数据库里**同时按 语义 + 时间 + 重要性** 三方面打分，返回**最相关、最新、最重要**的 K 条记忆。

```python
def hybrid_memory_retrieval(
     query: str,
     user_id: str,
     top_k: int = 5,
     alpha: float = 0.4,  # 语义相关性权重
     beta: float = 0.3,   # 时间新鲜度权重
     gamma: float = 0.3   # 重要性权重
 ) -> list:
     """
     混合检索：语义相关性 + 时间新鲜度 + 重要性
     """
     # 1. 按user_id过滤，获取候选记忆（召回更多，后面重排）
     candidates = milvus_client.search(
         collection_name="agent_memory",
         data=[get_embedding(query)],
         filter=f"user_id == '{user_id}' && is_deleted == false",
         limit=top_k * 3,  # 先召回3倍，后面重排
         output_fields=["memory_id", "content", "created_at", "importance_score"]
     )

     current_time = time.time()
     DECAY_RATE = 0.995  # 时间衰减率

     scored_memories = []
     for mem, semantic_score in zip(candidates[0], candidates[1]):
         # 时间新鲜度（基于Generative Agents的衰减公式）
         hours_passed = (current_time - mem["created_at"]) / 3600
         recency_score = DECAY_RATE ** hours_passed

         # 重要性归一化
         importance_score = mem["importance_score"] / 10.0

         # 综合评分
         final_score = (
             alpha * semantic_score +
             beta * recency_score +
             gamma * importance_score
         )
         scored_memories.append((mem, final_score))

     # 按综合评分排序，取Top K
     scored_memories.sort(key=lambda x: x[1], reverse=True)
     return [m for m, _ in scored_memories[:top_k]]
```

**艾宾浩斯遗忘机制**

```python
def update_memory_strength(memory_id: str, was_retrieved: bool):
     """
     每次检索后更新记忆强度：
     - 被成功检索并使用：重置衰减计时器（相当于一次"复习"）
     - 长期未被检索：按遗忘曲线降低重要性评分
     """
     mem = milvus_client.get(memory_id)

     if was_retrieved:
         # 复习效应：重置最后访问时间，小幅提升重要性
         milvus_client.update(memory_id, {
             "last_accessed": time.time(),
             "importance_score": min(10.0, mem["importance_score"] + 0.5)
         })
     else:
         # 遗忘效应：每天衰减5%，低于阈值则软删除
         days_since_access = (time.time() - mem["last_accessed"]) / 86400
         decay_factor = 0.95 ** days_since_access
         new_importance = mem["importance_score"] * decay_factor

         if new_importance < 1.0:
             # 重要性过低，标记为软删除
             milvus_client.update(memory_id, {"is_deleted": True})
         else:
             milvus_client.update(memory_id, {"importance_score": new_importance})
```

## **记忆检索：如何把记忆注入 Prompt**

```python
async def retrieve_memory(query: str, user_id: str) -> str:
    """
    检索与当前问题相关的历史记忆，返回格式化的Prompt片段
    """
    # 第一步：向量检索用户相关记忆（先过滤user_id，再做语义检索）
    relevant_memories = memory.search(
        query=query,
        user_id=user_id,
        limit=5
    )

    if not relevant_memories:
        return ""

    # 第二步：格式化为Prompt片段
    memory_context = "\n".join([
        f"- {m['memory']} (记录于{m['created_at']})"
        for m in relevant_memories
    ])

    return f"""
【用户历史信息】
以下是该用户的历史偏好和关键信息，请在回答时参考：
{memory_context}
"""

# 在对话流程中注入
async def chat(user_message: str, user_id: str, session_id: str):
    # 检索相关记忆
    memory_context = await retrieve_memory(user_message, user_id)

    # 构建完整的系统Prompt
    system_prompt = BASE_SYSTEM_PROMPT
    if memory_context:
        system_prompt += "\n\n" + memory_context

    # 调用LLM
    response = await llm.ainvoke(
        messages=[
            {"role": "system", "content": system_prompt},
            *short_term_memory.get_context(),
            {"role": "user", "content": user_message}
        ]
    )

    # 更新短期记忆
    short_term_memory.add_message("user", user_message)
    short_term_memory.add_message("assistant", response.content)

    return response.content
```

第一，检索时一定要先按 `user_id` 过滤，再做向量相似度计算，不能把所有用户的记忆混在一起检索。不然不同用户的信息会相互污染，也是严重的隐私问题。

第二，`limit=5` 不是固定值，要根据 Prompt 长度预算来决定。如果上下文窗口紧张，可以减少到 3 条；如果问题复杂、需要更多背景信息，可以增加到 8 条。

第三，检索到的记忆要带上时间戳，让 LLM 知道这条信息是什么时候记录的，方便它判断信息的时效性。![截屏2026-04-08 21.16.05](/Users/anji/Library/Application Support/typora-user-images/截屏2026-04-08 21.16.05.png)

## **四步记忆管理流程**

把上面所有内容串起来，完整的记忆管理流程是四步：

**第一步：Define（定义）**

明确哪些信息值得记忆。不是所有对话内容都有保存价值，要聚焦在：

- 用户明确表达的偏好（风险偏好、产品偏好、服务偏好）
- 用户的基本信息（企业规模、行业、注册地）
- 用户的重要决策和关键事件
- 对后续服务有实质参考价值的信息

"今天天气真好"不值得记忆，"我们公司明年计划做出口业务"值得记忆。

**第二步：Write（写入）**

用 LLM 从对话中提取记忆点，按类型分类后写入对应的存储：

- 情节记忆：写入向量数据库，绑定 user_id，附上重要性评分
- 语义记忆：写入共享知识库
- 程序性记忆：更新系统 Prompt 配置

**第三步：Manage（管理）**

写入前做冲突检测，决定 ADD/UPDATE/DELETE/NOOP 操作：

- 语义相似度 > 0.85 且内容有变化：UPDATE
- 新信息表明旧记忆已失效：DELETE
- 新信息之前没有类似记录：ADD
- 新信息与现有记忆一致：NOOP

同时运行 TTL 清理和遗忘机制，保持记忆库在合理规模。

**第四步：Read（读取）**

每次对话开始时：

1. 先检索长期记忆，用混合检索公式（语义 + 时间 + 重要性）取综合评分最高的 5 条
2. 格式化后注入系统 Prompt
3. 拼接短期记忆（最近 N 轮对话）
4. 发送给 LLM 生成回答
5. 对话结束后，更新被使用记忆的强度

这四步构成一个闭环，每次对话既消费记忆，又产生新的记忆，同时持续更新记忆的强度和时效性。

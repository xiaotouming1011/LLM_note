#### 1.Agent 的记忆分哪三种类型？各自用什么存储方式？

语义记忆：通用知识（产品条款/政策/FAQ），对所有用户共享，存共享向量数据库，按内容语义检索

情节记忆：用户专属历史信息（偏好/资质/历史决策），按 user_id 隔离存 Milvus，检索时先过滤 user_id 再语义检索

程序性记忆：操作流程和行为规则（'涉及高净值客户优先推荐私行产品'），直接固化为 System Prompt，不需要检索

##### 每次对话开始时，先按用户 ID 过滤，再做向量相似度检索，取相关度最高的 5 条，格式化后注入系统 Prompt

#### 2.记忆管理四步流程

短期记忆：对话上下文窗口，保留N轮对话

长期记忆：向量数据库分区储存。用 Milvus，存储结构上，按记忆类型和用户 ID 分区：

语义记忆分区：`user_id` 字段为空或固定值 `"global"`，对所有用户共享 情节记忆分区；`user_id` 字段绑定具体用户，检索时必须带 `user_id` 过滤条件

对话结束时的记忆提取：一次对话结束后，系统需要从对话内容里提取值得长期保存的记忆点，参考下面Mem0框架



Mem0框架：四种记忆操作，核心机制是：每次写入新记忆前，先用 LLM 比较新信息和现有记忆的语义关系，然后决定做哪种操作。

**记忆冲突怎么处理**

**方案一：语义去重判断**

用 LLM 比较新旧信息的语义相似度，相似度超过阈值则判断为同一实体的更新

```python
# 记忆冲突检查逻辑（异步版）
用于判断新记忆是 ADD 新增，还是 UPDATE 更新


async def check_memory_conflict(
    new_memory: str,
    existing_memories: list,
    similarity_threshold: float = 0.85
) -> dict:
    """
    检查新记忆是否与现有记忆冲突
    返回：{"action": "ADD/UPDATE", "conflict_memory_id": str or None}
    """
    # 无历史记忆 → 直接新增
    if not existing_memories:
        return {"action": "ADD", "conflict_memory_id": None}

    # 生成新记忆的向量
    new_embedding = await embedder.aembed_query(new_memory)

    # 遍历所有现有记忆，计算语义相似度
    for mem in existing_memories:
        similarity = cosine_similarity(new_embedding, mem["embedding"])

        # 相似度超过阈值 → 判定为同一条记忆
        if similarity > similarity_threshold:
            # 调用 LLM 二次确认：是否属于内容更新
            is_update = await llm_confirm_update(new_memory, mem["content"])
            if is_update:
                return {"action": "UPDATE", "conflict_memory_id": mem["memory_id"]}

    # 无冲突/无相似 → 新增
    return {"action": "ADD", "conflict_memory_id": None}
```

## **记忆检索：如何把记忆注入 Prompt**

```python
async def retrieve_memory(query: str, user_id: str) -> str:
    """
    检索与当前问题相关的历史记忆，返回格式化的Prompt片段
    """
    # 第一步：向量检索用户相关记忆（先过滤user_id，再做语义检索）
    relevant_memories = memory.search(
        query=query,
        user_id=user_id,
        limit=5
    )

    if not relevant_memories:
        return ""

    # 第二步：格式化为Prompt片段
    memory_context = "\n".join([
        f"- {m['memory']} (记录于{m['created_at']})"
        for m in relevant_memories
    ])

    return f"""
【用户历史信息】
以下是该用户的历史偏好和关键信息，请在回答时参考：
{memory_context}
"""

# 在对话流程中注入
async def chat(user_message: str, user_id: str, session_id: str):
    # 检索相关记忆
    memory_context = await retrieve_memory(user_message, user_id)

    # 构建完整的系统Prompt
    system_prompt = BASE_SYSTEM_PROMPT
    if memory_context:
        system_prompt += "\n\n" + memory_context

    # 调用LLM
    response = await llm.ainvoke(
        messages=[
            {"role": "system", "content": system_prompt},
            *short_term_memory.get_context(),
            {"role": "user", "content": user_message}
        ]
    )

    # 更新短期记忆
    short_term_memory.add_message("user", user_message)
    short_term_memory.add_message("assistant", response.content)

    return response.content
```

第一，检索时一定要先按 `user_id` 过滤，再做向量相似度计算，不能把所有用户的记忆混在一起检索。不然不同用户的信息会相互污染，也是严重的隐私问题。

第二，`limit=5` 不是固定值，要根据 Prompt 长度预算来决定。如果上下文窗口紧张，可以减少到 3 条；如果问题复杂、需要更多背景信息，可以增加到 8 条。

第三，检索到的记忆要带上时间戳，让 LLM 知道这条信息是什么时候记录的，方便它判断信息的时效性。![截屏2026-04-08 21.16.05](/Users/anji/Library/Application Support/typora-user-images/截屏2026-04-08 21.16.05.png)
